01 · Zihniyet

Sahne parçası değil, doküman ağacı

UGUI'de her buton, panel, ikon; Hierarchy'de gerçek bir GameObject'tir. Sahneye aittir, Transform'u vardır, Instantiate ile çoğaltılır. UI Toolkit'te ise eleman ağacı tamamen kendi dünyasında yaşar — UIDocument component'i bu ağacı sahneye "bağlar", ama elemanların kendisi asla GameObject olmaz.

Pratik sonucu: yüzlerce envanter slotu, tag butonu, diyalog satırı gibi tekrarlayan UI'larda UI Toolkit ciddi şekilde daha hafif çalışır — çünkü ortada GameObject instantiate/destroy maliyeti, ekstra Transform hesaplaması, Canvas rebuild'i yok.

UGUI zihniyeti

"UI = Sahne." Hierarchy penceresinde gezinirsin, her eleman Inspector'da ayrı ayrı Component taşır, Prefab olarak paketlenir.

UI Toolkit zihniyeti

"UI = Doküman." UXML'de yapıyı, USS'te stili tanımlarsın; C# tarafında sadece Q<T>() ile sorgu atarsın — web geliştirmedeki DOM/CSS ayrımının aynısı.

02 · Yapı

Hierarchy kurmak yerine UXML yazmak

UGUI'de bir panel + buton kurmak için Hierarchy'de sağ tık → GameObject oluşturursun. UI Toolkit'te bunun karşılığı bir .uxml dosyası; UI Builder penceresi de bunu görsel olarak düzenlemeni sağlar (Window → UI Toolkit → UI Builder).

Hierarchy (UGUI)
// Hierarchy penceresinde:
Panel (Image)
 └── PlayButton (Button, Image)
      └── Text (TextMeshProUGUI)

Her satır ayrı bir GameObject. Prefab yapıp variant türetmediğin sürece yeniden kullanım zahmetli.

MainMenu.uxml
<ui:VisualElement class="panel">
    <ui:Button
        name="play-button"
        text="Play"
        class="play-btn" />
</ui:VisualElement>

name = obje ismin gibi (tek, benzersiz sorgu için), class = stil paylaşımı için. Aynı UXML'i farklı sahnelerde tekrar kullanabilirsin.

03 · Görünüm

Inspector'da tek tek ayar yerine paylaşılan stil

UGUI'de bir butonun rengini, boyutunu, hover davranışını Inspector'dan o objeye özel ayarlarsın — Selectable'ın Normal/Highlighted/Pressed renkleri buna örnek. USS'te CSS mantığıyla class üzerinden tanımlarsın; :hover, :active, :focus pseudo-class'ları bile var.

Inspector ayarları
// Button (Script) → Transition: Color Tint
Normal Color:      #C83C3C
Highlighted Color: #DC5050
Pressed Color:     #A82E2E
// Image (Script)
Padding: 10, 20, 10, 20

Değişiklik yapmak için o objeye tıklayıp Inspector'ı düzenlersin. 50 tane benzer buton varsa 50 kere.

MainMenu.uss
.play-btn {
    background-color: rgb(200, 60, 60);
    padding: 10px 20px;
    border-radius: 8px;
    font-size: 24px;
}

.play-btn:hover {
    background-color: rgb(220, 80, 80);
}

Bir class'ı değiştirirsin, o class'a sahip her eleman güncellenir — 50 buton olsa da tek satır.

Not: DOTween'i doğrudan USS elemanlarına bağlayamazsın; onun yerine native transition-property / transition-duration gibi CSS geçişlerini ya da experimental.animation API'sini kullanırsın.
04 · Yerleşim

Anchor/Pivot yerine Flexbox

Bu muhtemelen en büyük alışkanlık değişimi. UGUI'de RectTransform'un anchor'ları, pivot'u, anchoredPosition'ı ile uğraşırsın; responsive için Vertical/Horizontal Layout Group + Content Size Fitter eklersin. UI Toolkit'in layout motoru Yoga (Flexbox'ın kendisi) — anchor kavramı yok, her şey flex kurallarıyla akar.

şekil 2 — konumu sen mi hesaplıyorsun, motor mu? anchoredPosition ↔ flex-direction/justify-content
Layout Group + Content Size Fitter
// Vertical Layout Group
Child Alignment: Middle Center
Spacing: 12
Control Child Size: ✓ Width / Height

// Content Size Fitter
Horizontal Fit: Preferred Size

İki ayrı Component eklemen, aralarındaki etkileşimi öğrenmen gerekir (biri boyutu, diğeri diziliş yönünü kontrol eder).

USS flex kuralları
.container {
    flex-direction: column;
    justify-content: center;
    align-items: center;
    flex-grow: 1;
}

flex-direction: row = Horizontal Layout Group, column = Vertical Layout Group. flex-grow, Content Size Fitter'ın işini tek satırda halleder.

05 · Kod

GetComponent yerine Query (Q<T>)

UGUI'de referansı Inspector'dan sürükleyip bırakırsın ya da GetComponent/GetComponentInChildren çağırırsın. UI Toolkit'te UIDocument'ın rootVisualElement'ından Q<T>("isim") ile ağaçta arama yaparsın — mantığı GetComponentInChildren ile birebir aynı, sözdizimi farklı.

MainMenu.cs (UGUI)
public class MainMenu : MonoBehaviour
{
    public Button playButton;

    void Start()
    {
        playButton.onClick.AddListener(OnPlayClicked);
    }

    void OnPlayClicked()
    {
        Debug.Log("Play basıldı");
    }
}
MainMenu.cs (UI Toolkit)
public class MainMenu : MonoBehaviour
{
    private VisualElement root;

    void OnEnable()
    {
        root = GetComponent<UIDocument>().rootVisualElement;
        var playButton = root.Q<Button>("play-button");
        playButton.clicked += OnPlayClicked;
    }

    void OnPlayClicked()
    {
        Debug.Log("Play basıldı");
    }
}

Q<Button>("play-button") UXML'deki name="play-button"'a karşılık gelir. Class ile de arama yapılır: root.Query<VisualElement>(className: "panel") — birden fazla sonuç döner, ToList() ile toplarsın.

06 · Referans tutma

Inspector'a sürükleme yok — bir kere sorgula, sakla

UGUI'de public Button playButton; yazıp Inspector'dan sürükleyip bırakırsın; referans sahne verisiyle birlikte serialize edilir. UI Toolkit'te VisualElement'lar Unity Object'i olmadığı için Inspector'a sürüklenemez — tek yol, ağacı bir kere Q<T>() ile sorgulayıp sonucu bir C# alanında (field) saklamak. Bunu genelde OnEnable'da, "kur ve sakla" mantığıyla yaparsın; her Update'te tekrar sorgulamazsın çünkü Query de bir maliyettir.

Referans + aç/kapat (UGUI)
public class ShopPanel : MonoBehaviour
{
    // Inspector'dan sürükle-bırak
    public Button buyButton;
    public GameObject lockedIcon;

    void Refresh(bool canAfford)
    {
        buyButton.interactable = canAfford;
        lockedIcon.SetActive(!canAfford);
        gameObject.SetActive(true); // panelin kendisi
    }
}

Referans Inspector'da sabit; interactable tıklanabilirliği, SetActive ise objeyi hierarchy'den tamamen kaldırır (Transform dahil her şeyi devre dışı bırakır).

Referans + aç/kapat (UI Toolkit)
public class ShopPanel : MonoBehaviour
{
    private Button buyButton;
    private VisualElement lockedIcon;
    private VisualElement panel;

    void OnEnable()
    {
        var root = GetComponent<UIDocument>().rootVisualElement;
        // bir kere sorgula, field'a sakla
        buyButton  = root.Q<Button>("buy-button");
        lockedIcon = root.Q<VisualElement>("locked-icon");
        panel      = root.Q<VisualElement>("shop-panel");
    }

    void Refresh(bool canAfford)
    {
        buyButton.SetEnabled(canAfford);
        lockedIcon.style.display = canAfford
            ? DisplayStyle.None : DisplayStyle.Flex;
        panel.style.display = DisplayStyle.Flex;
    }
}

Sorguyu OnEnable'da bir kere yapıp field'da tutmak, UGUI'deki Inspector referansının karşılığı. SetEnabled tıklanabilirliği, style.display görünürlüğü kontrol eder.

Ne yapmak istiyorsunUGUIUI Toolkit
Tamamen gizle, layout'tan da çıkar gameObject.SetActive(false) element.style.display = DisplayStyle.None
Görünmez yap ama yerini koru (layout bozulmasın) CanvasGroup.alpha = 0 element.style.visibility = Visibility.Hidden
Görünsün ama tıklanmasın (soluk/disabled görünüm) button.interactable = false button.SetEnabled(false)
Fare/tıklamayı yok say ama görsel olarak aktif kalsın CanvasGroup.blocksRaycasts = false element.pickingMode = PickingMode.Ignore
Dikkat: DisplayStyle.None verdiğin eleman layout hesabından tamamen çıkar (RectTransform'suz bir SetActive(false) gibi düşün) — o alanı boş bırakmasını istiyorsan Visibility.Hidden kullanmalısın, o da UGUI'deki CanvasGroup.alpha = 0'a karşılık gelir.
Birden fazla eleman için referans: Tag sistemindeki gibi çok sayıda benzer eleman varsa tek tek field açmak yerine bir Dictionary<TagData, VisualElement> ya da List<Button> tutup Query<Button>(className: "tag-btn").ToList() ile toplu sorgulayabilirsin — UGUI'deki GetComponentsInChildren<Button>() ile aynı mantık.
07 · Girdi sistemi

EventSystem GameObject'ine gerek yok

UGUI gereksinimi

Sahnede mutlaka bir EventSystem GameObject'i olmalı (+ Input Module). Yoksa hiçbir buton tıklamaz, Inspector'da sessizce başarısız olur.

UI Toolkit davranışı

UIDocument component'i runtime girdi işlemeyi kendi içinde barındırır, ayrı bir EventSystem gerekmez. Panel Settings asset'i sort order / scale mode gibi ayarları üstlenir — kabaca Canvas'ın karşılığı.

08 · Tekrarlayan elemanlar

Instantiate yerine CloneTree / ListView

Envanter slotu, tag butonu gibi tekrar eden bir öğeyi UGUI'de prefab'layıp Instantiate() edersin. UI Toolkit'te iki yol var: UXML'i bir VisualTreeAsset olarak yükleyip CloneTree() ile çoğaltmak, ya da veri listesi için hazır ListView kullanmak.

Prefab çoğaltma
foreach (var tag in allTags)
{
    var slot = Instantiate(tagPrefab, container);
    slot.GetComponent<TagSlot>().Bind(tag);
}
CloneTree çoğaltma
foreach (var tag in allTags)
{
    VisualElement slot = tagTemplate.CloneTree();
    slot.Q<Label>("tag-name").text = tag.displayName;
    container.Add(slot);
}
IndieGameDevTycoon bağlantısı: senin ScriptableObject-per-tag yapındaki her tag butonunu ayrı GameObject yapmak yerine, tek bir UXML template'ini CloneTree() ile çoğaltıp "selected" USS class'ını toggle ederek state yönetebilirsin — DOTween yerine native transition-property ile geçiş animasyonu da eklenir.
